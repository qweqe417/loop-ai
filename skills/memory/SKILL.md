# /memory — 记忆沉淀与清理

从会话中提取可复用经验，写入 `.claude/rules/loop-memory-*.md`。MEMORY 阶段自动调用（只做沉淀），总行数超过阈值自动触发清理。

手动触发：
- `/memory` — 沉淀当前会话经验
- `/memory --cleanup` — 只做清理，不写新内容

---

## 一、文件结构

```
.claude/rules/
  loop-memory-dev-pitfalls.md      ← 编码踩坑（框架陷阱、并发、边界条件等）
  loop-memory-dev-prohibited.md    ← 禁止操作（Guard 拦截、不可逆操作）
  loop-memory-dev-rules.md         ← 通用规则（架构决策、模块边界、代码风格）
  loop-memory-test-pitfalls.md     ← 测试踩坑（数据准备、Mock 遗漏、断言不准）
  loop-memory-test-patterns.md     ← 验证模式（场景设计、边界覆盖、回归策略）
  loop-memory-failures.md          ← 失败模式+修复路径（跨开发/测试）
```

## 二、沉淀（每次 MEMORY 阶段执行）

### Step 1: 读取现有记忆

先读 `loop-memory-*.md` **所有文件**，逐条过一遍标题和触发条件。

### Step 1.5: 逐条查重（写入前必须执行）

**在写每条新记忆之前**，搜索已有文件中是否存在相同根因的条目：

```
搜索关键词: 新条目的标题 + 触发条件中的框架/工具名 + 症状关键词
匹配规则:
  - 标题/症状高度相似 → 已有，跳过，不重复写
  - 同一根因的新表现 → 合并到已有条目，补充症状描述
  - 同一根因但修复方法不同（如框架升级后改用新方案） → 更新已有条目，标注版本
  - 完全不同 → 写入新条目
```

如果跳过：在摘要里说明 `跳过: [已有条目标题] 已覆盖`。

### Step 2: 判断什么值得记

**开发维度：**

| 信号 | → 分类 |
|------|--------|
| 修复 ≥2 次才通过的 bug | `dev-pitfalls` |
| Guard 拦截的操作 | `dev-prohibited` |
| 框架/库的非文档化行为陷阱 | `dev-pitfalls` |
| 并发、事务、缓存需特定序列才复现 | `dev-pitfalls` |
| 架构选择导致返工 | `dev-rules` |

**测试维度：**

| 信号 | → 分类 |
|------|--------|
| 场景断言误判（expected 写错） | `test-pitfalls` |
| 测试数据不足导致不稳定 | `test-pitfalls` |
| Mock 遗漏导致生产例外 | `test-pitfalls` |
| VERIFY→REPAIR→VERIFY 循环 ≥2 轮 | `test-patterns` |
| 验证流程某一步反复卡住 | `test-patterns` |

**跨维度：**

| 信号 | → 分类 |
|------|--------|
| 反复出现的同类失败+已知修复路径 | `failures` |
| 可归纳为模式的 bug（同类代码、同类根因） | `failures` |

**不记：** 拼写错误、查文档能修的、环境问题（服务没启）、跟已有重复的。

### Step 3: 写入格式

```markdown
### {一句话标题}
- **触发**: {具体场景，不是泛泛的 "阶段=repair"，而是 "当 X 且 Y 时"}
- **症状**: {具体错误信息和异常表现}
- **修复**: {可执行的具体动作，不是 "检查日志修正代码"}
```

**正确 vs 错误：**

```markdown
❌:
### 修复失败
- 触发: 阶段=repair, 类别=code_logic
- 症状: 代码逻辑错误导致失败
- 修复: 检查错误日志，修正代码逻辑

✅:
### Django ORM select_for_update 死锁
- 触发: 事务中对多表关联键使用 select_for_update，高并发下
- 症状: MySQL 报 Deadlock found when trying to get lock
- 修复: 按主键顺序加锁，外键查询放在事务外层，避免隐式跨表锁
```

新条目写入文件**顶部**，不覆盖已有。

### Step 4: 文件拆分与膨胀控制

**硬限制：**
- 每个类别最多 **2 个文件**（主文件 + 1 个溢出文件）
- 总文件数 **不超过 12 个**（6 类别 × 2）
- 每个文件 **≤ 200 行**

**拆分规则：**
- 新条目写入主文件顶部
- 主文件 > 200 行 → 截取底部最旧的条目，移到 `-2.md`
- `-2.md` 也 > 200 行 → **触发该类别强制清理**，不允许创建 `-3.md`

**强制清理规则：**
当某个类别需要第三个文件时，说明这个类别积累了太多。执行：
1. 逐条审这个类别的所有条目（主 + -2）
2. 删除过时（代码已不存在）
3. 合并重复（同根因）
4. 删半年未遇到的冷条目
5. 目标：**合并回 1 个文件 ≤ 200 行**

### Step 5: 输出摘要

```
本次沉淀 N 条:
  开发: dev-pitfalls X | dev-prohibited X | dev-rules X
  测试: test-pitfalls X | test-patterns X
  跨维度: failures X
  跳过: X 条（不重复/不具体/一次性的）
```

没有就输出 `本次无值得沉淀的经验。`

## 三、清理（总行数 > 500 行时自动触发，或手动 `/memory --cleanup`）

### 触发条件（满足任一即触发）

- 总行数 > 500 行（不含空行和标题行）
- 总文件数 > 8 个
- 某个类别出现 `-2.md`（说明单一类别在膨胀）

### 清理流程

逐条审查。**核心原则：用代码存在性判断，不用时间判断。**

**Step 1: 定位代码**

从每条记忆的"触发"和"修复"中提取关键符号，Read 对应源文件：

```
记忆: "Django ORM select_for_update 死锁，修复: 按主键顺序加锁"
提取: select_for_update → Grep 项目中是否还有这个调用
      → models.py 是否还存在涉及 select_for_update 的代码
```

**Step 2: 按代码状态分类处理**

```
代码不存在了:
  ├─ 类/函数被删/重命名了    → 删
  ├─ 文件被删了               → 删
  ├─ 框架迁移了（Django→FastAPI）→ 删
  └─ API 改了，修复方法失效    → 删

代码还在但需要合并:
  ├─ 同根因多条（不同标题但相同触发+修复）  → 合并成 1 条，取最完整的
  └─ 不同症状相同修复                        → 合并，补充症状列表

代码还在但需要更新:
  └─ 修复方法涉及旧 API/配置               → 更新到当前版本

代码还在，判断不了:
  └─ 不确定是否该删                        → 保留，标 [待确认]
```

**Step 3: 不用的规则**

- ~~"半年没遇到"~~ — 不可靠，代码在就留
- ~~"最近还遇到吗"~~ — AI 无法确定，不如不看

### 不确定的列出来问人

```
清理完成，以下 3 条标记为待确认：

[待确认] Django ORM deadlock
  原因: 最近半年没遇到，但项目还在用 Django ORM + MySQL
  建议: 保留？删除？

[待确认] Redis 连接池耗尽
  原因: 配置已改为 50，代码里 max_connections 已显式设置
  建议: 保留？删除？

[待合并] N+1 查询相关 3 条
  原因: pitfall-1 / pitfall-7 / failure-3 都是 select_related/prefetch_related 遗漏
  建议: 合并为 1 条？
```

直接回复保留/删除/合并，我来改。

### 清理输出摘要

```
清理完成:
  文件: X 个 → Y 个
  条目: 保留 N 条 | 删除 N 条（代码不存在/半年未遇到）| 合并 N 组 → N 条
  待确认: N 条（已列出，等你决定）
```

## 四、审核

```bash
git diff .claude/rules/loop-memory-*.md
```

确认后 commit。不确定的条目随时可以改。

## 五、约束

- 不确定的 → 不记。宁可漏过不可滥记
- 写前查重 → 标题/触发条件相似的不重复写
- 清理依据 → 代码还在不在，不用"多久没遇到"判断
- 不确定该不该删 → 保留，标 [待确认] 问人
- 同一次会话的同类失败 → 合并成 1 条
- 改完用 git diff 确认改动范围
